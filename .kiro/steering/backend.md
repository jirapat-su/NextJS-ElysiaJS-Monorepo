# Backend - Elysia & Effect-TS (Hexagonal Architecture)

## Architecture: Hexagonal (Ports & Adapters)

Dependencies point inward. Outer layers depend on inner layers, never the reverse.

```
Infrastructure (Adapters) → Application (Use Cases) → Domain (Core)
```

**Design validation:** "If you can run your domain logic from tests without any infrastructure, your boundaries are correct."

### Dependency Rule

| Layer | Responsibility | Depends On |
|-------|---------------|------------|
| Domain | Business logic, entities, value objects, errors | Nothing (pure) |
| Application | Use cases, orchestration, port definitions | Domain only |
| Infrastructure | Adapters (HTTP, DB, cache, external) | Application + Domain |

```
Handler (Driver Adapter) → Service (Use Case) → Repository Port ← Repository Adapter (Driven Adapter)
     Elysia HTTP              Effect.gen           type/interface         Prisma + Effect
```

## File Structure (Hexagonal per Feature)

See `structure.md` — Backend section for the complete directory tree.

Summary of hexagonal layers per feature:

```
Where does it go?
├─ Pure business logic, no I/O, no framework  → domain/
├─ Contract/interface for external dependency  → ports/
├─ Orchestrates domain + has side effects      → [moduleName].service.ts (use case)
├─ Talks to database (Prisma)                  → [moduleName].repository.ts (driven adapter)
├─ HTTP route definition (Elysia)              → [moduleName].handler.ts (driver adapter)
└─ Shared across features                      → src/utils/ or src/libs/
```

## Domain Layer (`domain/`)

Pure business logic. **Zero external dependencies** — no Prisma, no Elysia, no Effect (except `Data.TaggedError` for error definitions).

### Domain Errors (`[featureName].errors.ts`)

```typescript
import { Data } from 'effect';
import { type ErrorMsg, createErrorFactory } from '@/libs/effect';

// Repository-level errors (infrastructure boundary)
export class UserRepositoryError extends Data.TaggedError(
  'Repository/User/Error'
)<ErrorMsg> {
  static new = createErrorFactory(this);
}

// Domain-level errors (business rule violations)
export class UserNotFoundError extends Data.TaggedError(
  'Domain/User/NotFound'
)<ErrorMsg> {
  static new = createErrorFactory(this);
}

export class UserAlreadyExistsError extends Data.TaggedError(
  'Domain/User/AlreadyExists'
)<ErrorMsg> {
  static new = createErrorFactory(this);
}

export class UserInvalidStateError extends Data.TaggedError(
  'Domain/User/InvalidState'
)<ErrorMsg> {
  static new = createErrorFactory(this);
}
```

**Error `_tag` naming convention:**

| Prefix | Layer | Example |
|--------|-------|---------|
| `Domain/` | Business rule violation | `Domain/User/NotFound` |
| `Repository/` | Data access failure | `Repository/User/Error` |
| `Service/` | Use case orchestration failure | `Service/User/ValidationFailed` |
| `External/` | Third-party service failure | `External/Payment/Timeout` |

### Domain Types (`[featureName].types.ts`)

Use Prisma generated types first — only write custom types for things Prisma doesn't provide (e.g. branded types, union literals, response DTOs).

```typescript
import type { User, Prisma } from '@prisma/client';

// ✅ Re-export frequently used Prisma types for this feature
export type UserEntity = User;
export type UserWhereInput = Prisma.UserWhereInput;
export type UserCreateInput = Prisma.UserCreateInput;
export type UserUpdateInput = Prisma.UserUpdateInput;

// ✅ Custom types only when Prisma doesn't provide them — branded types, domain-specific unions
export type UserStatus = 'active' | 'inactive' | 'suspended';

// ✅ Response DTO — mapped from entity for API response (not generated by Prisma)
export type UserResponse = {
  readonly id: string;
  readonly email: string;
  readonly display_name: string;
  readonly status: string;
  readonly created_at: string;
};

// ✅ Use case specific input that differs from Prisma input (optional)
export type CreateUserUseCaseInput = {
  readonly email: string;
  readonly displayName: string;
};
```

**Rules:**
- Use `User`, `Prisma.UserWhereInput`, `Prisma.UserCreateInput` from `@prisma/client` directly
- Re-export as aliases in `types.ts` so other features don't need to import from `@prisma/client` directly
- Only write new types for: Response DTOs, branded types, union literals that Prisma doesn't generate
- Never duplicate types that Prisma already provides

### Domain Rules (`[featureName].rules.ts`) — Optional

```typescript
import type { UserEntity, UserStatus } from './user.types';

/**
 * Business rule: user can only be deactivated if currently active
 */
export const canDeactivateUser = (user: UserEntity): boolean =>
  user.status === 'active' && !user.isDeleted;

/**
 * Business rule: validate allowed email domains
 */
export const isEmailDomainAllowed = (email: string): boolean => {
  const BLOCKED_DOMAINS = ['tempmail.com', 'throwaway.email'];
  const domain = email.split('@')[1]?.toLowerCase() ?? '';
  return !BLOCKED_DOMAINS.includes(domain);
};

/**
 * Business rule: map entity to response DTO
 */
export const toUserResponse = (user: UserEntity): UserResponse => ({
  id: user.id,
  email: user.email,
  display_name: user.displayName,
  status: user.status,
  created_at: user.createdAt.toISOString(),
});
```

## Ports Layer (`ports/`)

Contracts (types) that define how the application interacts with external systems. Ports live in the feature, adapters implement them.

### Driven Port — Repository (`[featureName].repository.port.ts`)

```typescript
import { Effect } from 'effect';
import type { User, Prisma } from '@prisma/client';
import type { UserRepositoryError } from '../domain/user.errors';

/**
 * Driven Port: User persistence contract
 * Uses Prisma generated types (User, Prisma.UserWhereInput) directly
 */
export type FindUserByIdPort = (
  id: string
) => Effect.Effect<User | null, UserRepositoryError>;

export type FindUsersPort = (props: {
  where: Prisma.UserWhereInput;
  skip: number;
  take: number;
}) => Effect.Effect<{ users: User[]; total: number }, UserRepositoryError>;

export type CreateUserPort = (
  input: Prisma.UserCreateInput
) => Effect.Effect<User, UserRepositoryError>;

export type UpdateUserPort = (
  id: string,
  input: Prisma.UserUpdateInput
) => Effect.Effect<User, UserRepositoryError>;

export type SoftDeleteUserPort = (
  id: string,
  deletedBy: string
) => Effect.Effect<User, UserRepositoryError>;
```

### Driven Port — External Service (optional)

```typescript
import { Effect } from 'effect';

/**

 */

  email: string;
  displayName: string;
```

## Application Layer — Use Case (`[moduleName].service.ts`)

Orchestrates domain logic via `Effect.gen`. Depends on domain types and port types. Does NOT import Prisma or Elysia directly.

```typescript
import { Effect } from 'effect';
import type { CreateUserUseCaseInput } from '../domain/user.types';
import { UserAlreadyExistsError } from '../domain/user.errors';
import { isEmailDomainAllowed, toUserResponse } from '../domain/user.rules';
import { createUserRepository } from './createUser.repository';
import { findUserByEmailRepository } from './createUser.repository';

type Props = {
  readonly input: CreateUserUseCaseInput;
  readonly createdBy: string;
};

export const createUserService = (props: Props) =>
  Effect.gen(function* () {
    // 1. Domain rule: validate email domain
    if (!isEmailDomainAllowed(props.input.email)) {
      return yield* Effect.fail(
        UserAlreadyExistsError.new('Email domain not allowed')()
      );
    }

    // 2. Port call: check existing user (via driven adapter)
    const existingUser = yield* findUserByEmailRepository(props.input.email);
    if (existingUser) {
      return yield* Effect.fail(
        UserAlreadyExistsError.new('User already exists')()
      );
    }

    // 3. Port call: persist new user (via driven adapter)
    const user = yield* createUserRepository({
      email: props.input.email,
      name: props.input.displayName,
      createdBy: props.createdBy,
    });

    // 4. Domain rule: map entity to response DTO
    return toUserResponse(user);
  });
```

### Service Rules

- Use `Effect.gen` for orchestration
- Call domain rules for business validation
- Call repository/port functions for I/O
- Map entities to response DTOs using domain rules
- Never import Prisma or Elysia directly
- Cache only aggregated/computed data

## Infrastructure Layer — Driven Adapter (`[moduleName].repository.ts`)

Implements port contracts using Prisma + Effect. This is the only layer that touches the database.

```typescript
import { Effect, Data } from 'effect';
import type { Prisma } from '@prisma/client';
import { prisma } from '@/libs/prisma';
import { UserRepositoryError } from '../domain/user.errors';
import type { CreateUserPort, FindUserByIdPort } from '../ports/user.repository.port';

/**
 * Driven Adapter: Prisma implementation of FindUserByIdPort
 */
export const findUserByIdRepository: FindUserByIdPort = (id) =>
  Effect.tryPromise({
    try: () =>
      prisma.user.findFirst({
        where: { id, isDeleted: false },
      }),
    catch: (e) => UserRepositoryError.new('Failed to find user')(e),
  });

/**
 * Driven Adapter: Prisma implementation of CreateUserPort
 */
export const createUserRepository: CreateUserPort = (input) =>
  Effect.tryPromise({
    try: () =>
      prisma.user.create({
        data: {
          email: input.email,
          displayName: input.displayName,
          createdBy: input.createdBy,
        },
      }),
    catch: (e) => UserRepositoryError.new('Failed to create user')(e),
  });

/**
 * Helper: find user by email (not a port, internal to this adapter)
 */
export const findUserByEmailRepository = (email: string) =>
  Effect.tryPromise({
    try: () =>
      prisma.user.findFirst({
        where: { email, isDeleted: false },
      }),
    catch: (e) => UserRepositoryError.new('Failed to find user by email')(e),
  });
```

### Repository with Caching

```typescript
import { createCache } from '@/libs/cache';

const userCache = createCache('user');
const CACHE_TTL = { SHORT: 30_000, MEDIUM: 60_000, LONG: 300_000 };

export const listUsersRepository = (props: {
  where: Prisma.UserWhereInput;
  skip: number;
  take: number;
}) =>
  Effect.gen(function* () {
    const cacheKey = JSON.stringify(['list', props.skip, props.take, props.where]);

    const [total, users] = yield* userCache.getOrSet(
      cacheKey,
      Effect.all(
        [
          Effect.tryPromise({
            try: () => prisma.user.count({ where: props.where }),
            catch: (e) => UserRepositoryError.new('Failed to count')(e),
          }),
          Effect.tryPromise({
            try: () =>
              prisma.user.findMany({
                where: props.where,
                skip: props.skip,
                take: props.take,
              }),
            catch: (e) => UserRepositoryError.new('Failed to fetch')(e),
          }),
        ],
        { concurrency: 'unbounded' }
      ),
      CACHE_TTL.MEDIUM
    );

    return { users, total };
  });
```

### Write with Cache Invalidation

```typescript
export const updateUserRepository = (id: string, data: UpdateUserInput & { updatedBy: string }) =>
  Effect.gen(function* () {
    const user = yield* Effect.tryPromise({
      try: () =>
        prisma.user.update({
          where: { id },
          data: { ...data, updatedAt: new Date() },
        }),
      catch: (e) => UserRepositoryError.new('Failed to update')(e),
    });

    // Single key invalidation
    yield* userCache.delete(JSON.stringify(['profile', id]));

    // Batch key invalidation
    yield* userCache.delete([
      JSON.stringify(['profile', id]),
      JSON.stringify(['settings', id]),
    ]);

    // Clear all keys in namespace
    yield* userCache.clear();

    return user;
  });
```

### Cache Behavior Notes

- `getOrSet` skips caching `null` and `undefined` values — prevents caching empty query results that would never invalidate naturally
- `delete` accepts `string | string[]` for single or batch key invalidation
- All cache operations are infallible (never throw) — failures are logged and return fallback values
- Cache miss on Redis failure gracefully falls through to `compute`

## Infrastructure Layer — Driver Adapter (`[moduleName].handler.ts`)

Elysia HTTP route. Converts HTTP request → use case call → HTTP response. This is the outermost layer.

```typescript
import { Elysia } from 'elysia';
import { Effect } from 'effect';
import { RUNTIME } from '@/libs/effect';
import { API_RETRY_POLICY, API_TIMEOUT_POLICY } from '@/utils/apiPolicy';
import { ERROR_MESSAGE_SCHEMA } from '@/utils/schema';
import { logger } from '@/libs/logger';
import { authorizationPlugin } from '@/plugins/authentication';
import { createUserService } from './createUser.service';
import { CREATE_USER_BODY_SCHEMA, CREATE_USER_RESPONSE_SCHEMA } from './createUser.schema';

export const createUserHandler = new Elysia({ name: 'createUser.handler' })
  .use(authorizationPlugin)
  .post(
    '/',
    async ({ body, status, request, tokenPayload }) => {
      const serviceResult = Effect.gen(function* () {
        return yield* createUserService({
          input: body,
          createdBy: tokenPayload?.userId ?? '',
        });
      }).pipe(
        Effect.retry(API_RETRY_POLICY),
        Effect.timeout(API_TIMEOUT_POLICY),
        Effect.tapError((error) => {
          logger.error({ error, endpoint: request.url }, 'Create user error');
          return Effect.fail(error);
        })
      );

      return await Effect.match(serviceResult, {
        onFailure: (error) => {
          switch (error._tag) {
            case 'Domain/User/AlreadyExists':
              return status(409, { message: error.msg ?? 'User already exists' });
            case 'Repository/User/Error':
              return status(500, { message: error.msg ?? 'Database error' });
            default:
              return status(500, { message: 'Internal Server Error' });
          }
        },
        onSuccess: (data) => status(201, data),
      }).pipe(RUNTIME('user')().runPromise);
    },
    {
      body: CREATE_USER_BODY_SCHEMA,
      response: {
        201: CREATE_USER_RESPONSE_SCHEMA,
        409: ERROR_MESSAGE_SCHEMA,
        500: ERROR_MESSAGE_SCHEMA,
      },
      detail: {
        summary: 'Create user',
        description: 'Creates a new user account',
      },
    }
  );
```

### Handler Rules

- `Effect.match` with `{ onFailure, onSuccess }` is the standard pattern for mapping Effect results to HTTP responses
- `RUNTIME` is a project-specific singleton wrapper around `ManagedRuntime.make` — do not replace with generic Effect runtime
- `RUNTIME` uses feature name only: `RUNTIME('user')`, `RUNTIME('product')`
- Apply `Effect.retry(API_RETRY_POLICY)` + `Effect.timeout(API_TIMEOUT_POLICY)` on all service calls
- Map domain error `_tag` to appropriate HTTP status codes
- Log errors with `Effect.tapError` before matching

### Error `_tag` → HTTP Status Mapping

| Error `_tag` Pattern | HTTP Status | Description |
|----------------------|-------------|-------------|
| `Domain/*/NotFound` | 404 | Resource not found |
| `Domain/*/AlreadyExists` | 409 | Conflict / duplicate |
| `Domain/*/InvalidState` | 422 | Business rule violation |
| `Domain/*/Forbidden` | 403 | Authorization failure |
| `Repository/*/Error` | 500 | Database error |
| `External/*/Error` | 502 | Third-party service failure |
| `default` | 500 | Internal Server Error |

## Schema (`[moduleName].schema.ts`)

Zod schemas for OpenAPI generation. Never use `.transform()` — breaks JSON Schema.

```typescript
import { z } from 'zod';

// Request schemas
export const CREATE_USER_BODY_SCHEMA = z.object({
  email: z.string().email(),
  display_name: z.string().min(1).max(100),
});

// Response schemas
export const CREATE_USER_RESPONSE_SCHEMA = z.object({
  id: z.string(),
  email: z.string(),
  display_name: z.string(),
  status: z.enum(['active', 'inactive', 'suspended']),
  created_at: z.string(),
});

// Query schemas (no .transform!)
export const LIST_USERS_QUERY_SCHEMA = z.object({
  page: z.coerce.number().int().positive().default(1),
  limit: z.coerce.number().int().positive().max(100).default(10),
  search: z.string().optional(),
  isDeleted: z.enum(['true', 'false']).optional(),
});
```

## Full Hexagonal Flow (Summary)

```
HTTP Request
    ↓
[Handler] (Driver Adapter — Elysia)
    ↓ calls
[Service] (Use Case — Effect.gen)
    ↓ validates via          ↓ calls
[Domain Rules]          [Repository] (Driven Adapter — Prisma)
    (pure functions)         ↓
                        [Database]
    ↓ maps via
[Domain Types] → Response DTO
    ↓
HTTP Response
```

### Layer Import Rules

| From → To | Domain | Ports | Service | Repository | Handler | Schema |
|-----------|--------|-------|---------|------------|---------|--------|
| Domain | ✅ | ❌ | ❌ | ❌ | ❌ | ❌ |
| Ports | ✅ | ✅ | ❌ | ❌ | ❌ | ❌ |
| Service | ✅ | ✅ | ✅ | ✅ | ❌ | ❌ |
| Repository | ✅ | ✅ | ❌ | ✅ | ❌ | ❌ |
| Handler | ✅ | ❌ | ✅ | ❌ | ✅ | ✅ |

**Violations to catch:**
- Domain importing Prisma, Elysia, or libs
- Handler importing repository directly (bypass service)
- Service importing Elysia
- Repository importing handler or schema

## Anti-Patterns

| Anti-Pattern | Problem | Fix |
|--------------|---------|-----|
| Handler calls repository directly | Skips business logic | Always go through service |
| Domain imports Prisma types | Leaks infrastructure | Use domain types, map in repository |
| Service returns Prisma model directly | Couples response to DB schema | Map to domain response DTO |
| Business logic in handler | Handler becomes god object | Move to domain rules or service |
| Shared mutable state in service | Race conditions | Use Effect for concurrency |
| Generic error tags | Can't map to HTTP status | Use specific `_tag` per error type |
